# PLAN — aiMCPGate

Продуктовое видение MCP-шлюза `aiMCPGate`. Технический план и разбивка по
этапам — [`TECHNICAL_PLAN.md`](TECHNICAL_PLAN.md); за пределами MVP —
[`POST_MVP_PLAN.md`](POST_MVP_PLAN.md).

## 1. Что это

`aiMCPGate` — **шлюз / прокси для MCP-серверов** на Go. Он presents себя
клиенту (Claude Code, Cursor, любой MCP-совместимый агент) как **один**
MCP-сервер, а под капотом **мультиплексирует** вызовы к нескольким upstream
MCP-серверам и **логирует каждый вызов** инструмента/ресурса.

MCP (Model Context Protocol) — открытый протокол, которым агентские среды
подключаются к внешним инструментам и ресурсам. Экосистема быстро растёт в
2026: у активного пользователя MCP уже подключено несколько серверов
(filesystem, github, поиск, кастомные). Управлять ими по отдельности в конфиге
каждого клиента — неудобно. `aiMCPGate` даёт **одну точку входа** и **единый
журнал** того, что агенты реально вызывают.

Первоисточник по протоколу — <https://modelcontextprotocol.io> (официальная
спецификация; обязательна к прочтению перед Этапом 1 — см. TECHNICAL_PLAN §1).

## 2. Зачем (мотивация)

Соло pet-проект интроверта-разработчика. Две реальные ценности:

1. **Практическая.** Один эндпоинт вместо N в конфиге каждого клиента;
   включать/выключать upstream-серверы централизованно; **видеть в логе**, что
   именно и как часто вызывают агенты (аудит, отладка, понимание собственных
   паттернов работы).
2. **Обучающая / портфолио.** Go-конкурентность (мультиплексирование через
   горутины/каналы), работа с внешними процессами (`os/exec`, stdio),
   JSON-RPC 2.0, транспорты stdio и HTTP/SSE. Осознанно поднятая планка: не
   «личный скрипт», а аккуратный инструмент с тестами и документацией.

Приоритет — **Go**. Fable 5 не используется.

## 3. Ключевая ценность поверх «просто прокси»

Голый прокси бесполезен. Добавленная ценность:

- **Агрегация каталога.** Списки tools/resources всех upstream-серверов
  сливаются в один каталог с неймспейсингом (`<upstream>__<tool>`), чтобы имена
  не сталкивались.
- **Журнал вызовов.** Каждый прошедший через шлюз вызов пишется структурированно
  (какой upstream, метод, инструмент, длительность, успех/ошибка) — JSON-строки
  в файл/stdout. Это и есть «шлюз», а не «труба».
- **(Фаза 2) Политика доступа.** Какие инструменты какому клиенту видны — простой
  allow/deny поверх агрегированного каталога.

## 4. Кто пользователь

Один человек — сам разработчик. Запускается **локально** как дочерний процесс
клиента (stdio) или как локальный/VPS-сервис (HTTP/SSE, Фаза 2). Мультитенантности
нет. Аутентификации между клиентом и шлюзом в Фазе 1 нет (доверенный локальный
процесс); в Фазе 2 — минимальная (см. POST_MVP).

## 5. Границы MVP (двухфазная структура)

**Фаза 1 (ядро).**
- Мультиплексирование 2+ **stdio** upstream MCP-серверов за одним **stdio**
  эндпоинтом (тот же транспорт, что видит Claude Code).
- MCP-handshake (`initialize`), агрегация `tools/list` + `resources/list`,
  маршрутизация `tools/call` / чтения ресурсов к нужному upstream.
- Базовое логирование каждого вызова в файл/stdout.
- Конфиг: список upstream-серверов (команда + аргументы + env), путь к логу.

**Фаза 2 (расширение).**
- **HTTP/SSE** транспорт как альтернатива stdio (на стороне клиента).
- Поддержка **HTTP upstream-серверов** (не только stdio-процессов).
- Простой просмотрщик журнала: CLI-команда истории **или** минимальный веб-вью.
- Опционально — политика доступа (видимость инструментов по клиенту).

## 6. Явно вне MVP (см. POST_MVP_PLAN.md)

Горячая перезагрузка конфига, health-checks и авто-рестарт упавших upstream,
кэширование ответов, метрики/трейсинг, multi-client concurrency поверх stdio,
трансформация/фильтрация схем инструментов, вторичная аутентификация к upstream
за пределами передачи env.

## 7. Нефункциональные принципы

- **$0/мес** по умолчанию — локальный процесс, без внешних сервисов и без
  телеметрии. VPS — только если пользователь захочет держать HTTP-режим постоянно
  (обоснование в TECHNICAL_PLAN §Развёртывание).
- **Секреты не в git.** Токены upstream-серверов — через env/`.env` (в
  `.gitignore`), никогда не в конфиге под версионным контролем и никогда в логах.
- **Кросс-платформенность** (linux/darwin/windows) в рамках stdio/os.exec.
- **Тестируемость** — всё, кроме самого системного вызова процесса, за
  интерфейсами и покрыто тестами (см. SKILL.md).
