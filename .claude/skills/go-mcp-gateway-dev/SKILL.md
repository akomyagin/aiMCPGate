# SKILL: go-mcp-gateway-dev — конвенции проекта `aiMCPGate`

Конкретные конвенции **именно этого проекта** для написания Go-кода. Не общий
гайд «как писать Go», а специфика `aiMCPGate`. Применяй при реализации любого
этапа. Опорные документы: [`../../../docs/TECHNICAL_PLAN.md`](../../../docs/TECHNICAL_PLAN.md),
[`../../../docs/PLAN.md`](../../../docs/PLAN.md).

## 0. Спецификация MCP — сначала читать, потом кодить

**Главное правило проекта.** Детали протокола MCP (методы `initialize`,
`tools/list`, `tools/call`, форма ошибок, фрейминг stdio, SSE) **не выдумывать**.
До кода в Этапе 1 прочитать <https://modelcontextprotocol.io> и зафиксировать
выверенное в `docs/MCP_NOTES.md`. Пока деталь не сверена — помечать в коде/плане
`[TODO уточнить по официальной спецификации в Этапе 1]`, а не угадывать.

## 1. Структура: `cmd/` + `internal/`, почему нет `pkg/`

- `cmd/aimcpgate/main.go` — **тонкий**: собирает конфиг → логгер → registry →
  transport и блокирует до сигнала. Никакой бизнес-логики в `main`.
- Всё ядро — в `internal/`, чтобы компилятор **запрещал** внешний импорт: это
  приложение, а не библиотека. Пакеты: `config`, `registry`, `transport`,
  `logging` (+ `mcp`, `upstream` с Этапа 1–2).
- **`pkg/` не заводить.** Публичный API — только при реальном спросе.
- По файлу на cobra-команду в `internal/cli/` (`serve.go`, `logs.go`,
  `version.go`), общий каркас — в `root.go` (вводится, когда появятся команды,
  Этап 4+).

## 2. Upstream stdio-процессы: `os/exec` за интерфейсом

Как в `gitl` (git через `os/exec`) — здесь так же с upstream MCP-серверами.
Весь `os/exec` спрятан в `internal/upstream` (`stdioConn`).

- Запуск — `exec.CommandContext` (не `exec.Command`): контекст протянут от
  `signal.NotifyContext` → отмена по Ctrl-C завершает дочерние процессы.
- Пробросить `Env` upstream (секреты — из env/`.env`, **не** из конфига под git).
- Подключить `cmd.Stdin` (pipe для записи запросов) и `cmd.Stdout` (pipe для
  чтения ответов); `cmd.Stderr` — в операционный лог с префиксом имени upstream.
- При старте проверять доступность команды (`exec.LookPath`) и давать дружелюбную
  ошибку, а не панику; упавший upstream **изолируется**, не роняет шлюз.
- На завершении — корректно закрыть stdin, дождаться `cmd.Wait`, при таймауте —
  kill (по возможности — process group, чтобы не оставить сирот).
- Интерфейс `upstream.Conn` вводить **на второй реализации** (`httpConn`, Фаза 2),
  не раньше (правило «интерфейс на второй реализации»).

## 3. JSON-RPC 2.0: фрейминг и парсинг

- **stdio-фрейминг — line-delimited JSON** (одно JSON-RPC сообщение = одна строка,
  разделитель `\n`). Сверить точное правило со спецификацией
  `[TODO уточнить ... Этап 1]`. Читать `bufio.Scanner`/`bufio.Reader` **построчно**;
  **антипаттерн** — пытаться распарсить весь поток одним `json.Decoder` без учёта
  фрейминга или split по случайным разделителям.
- Типы — в `internal/mcp`: `Request{JSONRPC,ID,Method,Params}`,
  `Response{JSONRPC,ID,Result,Error}`, `Notification{JSONRPC,Method,Params}`,
  `Error{Code,Message,Data}`. `Params`/`Result` — `json.RawMessage`, чтобы
  **проксировать насквозь** неизвестные поля без потерь (свойство прозрачности
  шлюза важнее, чем строгая типизация каждого метода).
- `id` — может быть числом или строкой по спецификации: хранить как
  `json.RawMessage` или `any`, не хардкодить `int`.
- Запись в stdin upstream — **сериализована** (один writer-мьютекс/канал на
  upstream): конкурентная запись двух сообщений перемешает байты.

## 4. Мультиплексирование: горутины + каналы

- **Одна горутина-читатель на upstream** читает его stdout построчно, парсит
  сообщение и по `id` доставляет ответ ожидающему вызову через
  `map[id]chan Response` (под мьютексом). Уведомления (`id == null`) — в отдельный
  обработчик.
- **Разделение id-пространств:** id клиента ≠ id upstream. Шлюз генерирует свой
  upstream-side id (атомарный счётчик), ведёт сопоставление `upstreamID →
  (clientID, waiterChan)`; в ответе клиенту восстанавливает исходный clientID.
- **Fan-out на старте** — `initialize` + `tools/list` + `resources/list` ко всем
  enabled upstream параллельно через `golang.org/x/sync/errgroup`; сбор в общий
  каталог под мьютексом.
- **`context` протянут насквозь**; на каждый вызов — таймаут из конфига
  (`context.WithTimeout`), при истечении — снять waiter, вернуть error клиенту.
- Где есть конкурентность — тесты гонять с **`-race`** (обязательно).

## 5. Агрегация каталога и маршрутизация

- Неймспейсинг клиент-facing имён: `<upstream>__<tool>` (разделитель `__`;
  проверить валидность в спецификации `[TODO ... Этап 1]`).
- Таблица маршрутизации `namespacedName → (upstream, originalName)` в `registry`
  под мьютексом. При `tools/call` — резолв + переписывание `name` в оригинал
  перед форвардом.
- `description` и `inputSchema` инструментов — проксировать **как есть**
  (`json.RawMessage`): клиент должен видеть тот же контракт, что даёт upstream.
- Коллизии имён между upstream разрешаются неймспейсингом автоматически; логировать
  на debug факт коллизии оригинальных имён.

## 6. Журнал вызовов и секреты

- Журнал вызовов (`internal/logging.CallLog`) — **отдельно** от операционного
  `slog`: свой writer, **JSON-строки** (одна запись — строка), append в файл из
  конфига или stdout. Потокобезопасность — мьютекс на writer.
- `CallRecord`: `time, upstream, method, tool, client, duration, ok, error`
  (см. `internal/logging/logging.go`).
- **Секреты не логировать никогда** — ни `api_key`/токены upstream, ни `arguments`
  вызова целиком (могут содержать секреты). По умолчанию журнал пишет только имя
  инструмента и метаданные; логирование payload — отдельный opt-in (post-MVP) с
  маскированием. Тест должен проверять **отсутствие** секретов в журнале и stderr.

## 7. Тесты: детерминированные fake-процессы, не только моки

- **Fake upstream MCP-сервер** — маленький Go-бинарь в `testdata/` (или собранный
  из теста), говорящий по MCP через stdin/stdout: даёт **детерминированный
  интеграционный ярус**, гоняющий реальный `os/exec` + фрейминг, а не только
  мок-`io.Reader`. Это ключевая проверка мультиплексора.
- **Table-driven** — для JSON-RPC кодека (`internal/mcp`), неймспейсинга,
  разрешения коллизий, границ таймаута.
- **`httptest.Server`** — для HTTP-upstream и SSE-транспорта (Фаза 2).
- **`-race`** — везде, где есть горутины (мультиплексор, читатели upstream).
- End-to-end (Этап 4): fake-клиент + 2 fake-upstream, проверка агрегации,
  проксирования вызова, записи журнала и отсутствия секретов.

## 8. Общие правила

- Go **1.23**: `slices`, `maps`, `log/slog`, `errors.Join`, `errgroup`.
- Кросс-платформенность: `os.UserConfigDir()`/`filepath.Join`, не хардкодить пути.
- Операционные логи — `log/slog` (JSON, stderr); `--verbose` → debug.
- Различать ошибки типизированно (`errors.Is`/типизированные), не строковым
  сравнением; upstream-ошибку заворачивать в JSON-RPC error клиенту, не паниковать.
- Каждая заглушка `internal/*` из Этапа 0 помечена «Реализация — Этап N+»; при
  реализации **заменять** содержимое, а не плодить параллельные файлы.
- Интерфейс вводится **на второй реализации**, не на первой (анти-over-engineering).
