# CLAUDE.md

Инструкции для AI-сессий (Claude Code) в репозитории `aiMCPGate`. Эти инструкции
имеют приоритет над поведением по умолчанию — следуй им точно.

## Что это за репозиторий

`aiMCPGate` — **MCP-шлюз / прокси** на **Go**: presents себя MCP-клиенту
(Claude Code и др.) как один MCP-сервер, мультиплексирует вызовы к нескольким
upstream MCP-серверам (stdio-процессы и HTTP-эндпоинты), агрегирует их каталоги
tools/resources в один и логирует каждый вызов. Соло pet-проект с приоритетом
**обучения Go** (конкурентность, `os/exec`, JSON-RPC 2.0, транспорты stdio и
HTTP/SSE). Расходы — **$0/мес** по умолчанию (локальный процесс), без телеметрии.

Полное видение — [`docs/PLAN.md`](docs/PLAN.md); технический план и разбивка по
Этапам — [`docs/TECHNICAL_PLAN.md`](docs/TECHNICAL_PLAN.md); за пределами MVP —
[`docs/POST_MVP_PLAN.md`](docs/POST_MVP_PLAN.md).

## Структура репозитория

| Путь | Содержимое |
|---|---|
| `docs/` | `PLAN.md`, `TECHNICAL_PLAN.md`, `POST_MVP_PLAN.md` (и `MCP_NOTES.md` с Этапа 1) |
| `.claude/skills/go-mcp-gateway-dev/SKILL.md` | Конвенции именно этого проекта для написания кода |
| `cmd/aimcpgate/main.go` | Тонкий `main`: конфиг → логгер → registry → transport, блокировка до сигнала |
| `internal/config` | Парсинг+валидация YAML-конфига upstream-серверов, env-expansion секретов |
| `internal/registry` | Ядро мультиплексора: upstream-соединения, MCP-handshake, агрегированный каталог, маршрутизация |
| `internal/transport` | Клиент-facing сервер: stdio (Фаза 1), HTTP/SSE (Фаза 2) за общим `Server` |
| `internal/logging` | Операционный `slog` + `CallLog` — журнал вызовов (JSON-lines) |
| `internal/mcp`, `internal/upstream` | Вводятся с Этапа 1–2 (JSON-RPC слой; одно upstream-соединение) |

`pkg/` намеренно **нет** — всё ядро в `internal/`, чтобы никто не импортировал
незрелый API как библиотеку. Публичный API появится только при реальном спросе.

## Git / dev-workflow (обязательный процесс на каждый этап/задачу)

1. Opus 4.8 — если требуется детальное планирование этапа — планирование, затем написание кода.
2. Sonnet — проверка качества покрытия тестами, тестирование, проверка работоспособности, проверка покрытия новых функций.
3. Opus — независимое ревью: skill /code-review на diff ветки, фиксируем замечания.
4. Цикл исправлений — до 3 итераций: Sonnet правит замечания → тесты снова.
5. Commit + push + PR — conventional-commit с русским subject, PR в master-ветку.

Коммить/пуш только когда этап завершён (или пользователь явно попросил).

## Команды разработки

```bash
export PATH="$HOME/sdk/go/bin:$PATH"   # если go не в PATH
go build ./...                         # сборка (зелёная с Этапа 0)
go vet ./...
gofmt -l .                             # пусто = отформатировано
go test -race ./...                    # тесты; -race обязателен где есть конкурентность

go run ./cmd/aimcpgate serve --config ./config.yaml   # Этап 4+: stdio-шлюз
go run ./cmd/aimcpgate logs --tail 50                 # Этап 5+: журнал вызовов
go run ./cmd/aimcpgate version
```

Перед коммитом кода: `go build ./...` и `go test ./...` (с `-race` где есть
конкурентность) должны проходить; желателен `gofmt` и `go vet`.

## Ключевые архитектурные решения (не переоткрывай их без причины)

- **Прочитать спецификацию MCP** (<https://modelcontextprotocol.io>) **до кода в
  Этапе 1**; выверенные детали — в `docs/MCP_NOTES.md`. Не выдумывать детали
  протокола — помечать `[TODO уточнить по официальной спецификации в Этапе 1]`.
- Upstream stdio-серверы — системные процессы через **`os/exec`** (как в `gitl`),
  за интерфейсом в `internal/upstream`. Отмена — через `context` от
  `signal.NotifyContext`.
- **Один горутина-читатель на upstream**, демультиплекс ответов по JSON-RPC `id`;
  запись в stdin upstream **сериализована** (мьютекс/канал).
- Агрегация каталогов — неймспейсинг `<upstream>__<tool>` + таблица маршрутизации
  `namespacedName → (upstream, originalName)`; при `tools/call` имя переписывается
  обратно (TECHNICAL_PLAN §4.2–4.3).
- **Секреты (токены upstream) — через env/`.env`** (в `.gitignore`), **никогда**
  в конфиге под git и **никогда в логах** (ни операционных, ни в журнале вызовов).
- Правило зависимостей: «интерфейс появляется на второй реализации, не на первой»
  (напр. `upstream.Conn` — когда появляется `httpConn` в Фазе 2).
- Docker Compose — **не в Фазе 1** (шлюз = локальный stdio-процесс); опционально
  в Этапе 5 для постоянного HTTP-режима (TECHNICAL_PLAN §7).
